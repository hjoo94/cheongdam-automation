## 3. 서버 연동 개요

### 3.1 기본 URL

- **이언트 기본 서버 주소:** `app/main.js`의 `DEFAULT_SERVER_BASE_URL`  
  - 기본값: `http://43.203.124.132:4300`  
  - 구 IP `43.201.84.136` 등은 코드/설정에서 **자동 치환·안내 문구**로 언급될 수 있음. **실제 운영 Lightsail/Elastic IP**와 맞는지 배포 전에 반드시 확인.
- **서버 공개 베이스 URL:** `03_license_server/server.js`의 `PUBLIC_BASE_URL`  
  - 기본값: `http://43.203.124.132:4300`  
  - 업데이트 매니페스트의 다운로드 URL이 이 값을 기준으로 생성됨.

운영 서버 IP가 바뀌면 **서버 `.env`의 `PUBLIC_BASE_URL`**, **클라이언트가 사용하는 `settings.json`의 `serverBaseUrl`**, 필요 시 **`CHUNGDAM_SERVER_URL`** 을 일치시켜야 함.

### 3.2 서버 포트

- `PORT` 환경변수 (기본 `4300`), `HOST` (기본 `127.0.0.1` — 운영 시 `0.0.0.0` 등으로 바인딩하는 방식은 배포 스크립트/프로세스 관리에 따름).

### 3.3 헬스 체크

- `GET /health` → `{ ok: true, status: 'healthy' }` 형태로 서버 생존 확인 가능.

## 4. 라이선스 흐름 (핵심)

1. **클라이언트**는 사용자 데이터 경로에 `license.json` 등을 저장하고, 기동 시 **설정의 `serverBaseUrl`**(없으면 기본 URL)로 서버에 접속.
2. **서버**는 `POST /api/license/verify` 로 라이선스 키·기기 지문 등을 검증하고, 허용 시 기능·만료 정보를 내려줌. (구체 필드는 `server.js`의 `handleLicenseVerify` 및 클라이언트 `main.js`의 검증 로직 참고.)
3. **라이선스 데이터 파일:** 서버 측 `03_license_server/data/licenses.json` (백업 필수).
4. **관리자:** `04_license_admin` 앱이 `ADMIN_SECRET` 등으로 보호된 **관리 API**(`GET/POST /api/admin/licenses` 등)를 사용해 라이선스 CRUD·연장·비활성화.

**보안:** `03_license_server/.env`의 `ADMIN_SECRET`, OpenAI 키 등은 **저장소에 커밋하지 말 것.** 운영 서버에만 배치.

## 5. 자동 업데이트(패치) 흐름 — 사용자가 프로그램을 실행하면

1. 클라이언트(`app/main.js`)가 **업데이트 앱 ID** `UPDATE_APP_ID = 'client'` 로 최신 매니페스트를 요청:  
   `GET {serverBaseUrl}/api/updates/client/latest`
2. **서버**(`buildUpdateManifestResponse`)가 `UPDATE_MANIFEST.client`를 읽어 다음을 반환:
   - `version` — 예: `process.env.CLIENT_APP_VERSION` 또는 코드 폴백 **`1.0.30`**
   - `fileName` — 예: `process.env.CLIENT_APP_FILE` 또는 코드 폴백 **`Cheongdam Bot Setup 1.0.30.exe`**
   - `url` — `{PUBLIC_BASE_URL}/downloads/{fileName}` (URL 인코딩 적용)
   - `sha256` — 서버 디스크上的 `03_license_server/downloads/{fileName}` 파일이 존재할 때만 유효한 64자 해시, 없으면 빈 문자열·`installerReady: false`
3. 클라이언트가 **로컬 버전과 비교**해 서버 버전이 더 높으면, HTTPS 정책을 만족하는 경우 **해당 URL로 설치 파일 다운로드** 후 설치 래퍼 실행 등으로 업데이트 진행.

**운영 체크리스트 (패치가 사용자에게 실제로 적용되려면)**

| 순서 | 작업 |
|------|------|
| 1 | 로컬에서 `npm run build:win` 등으로 **실제 설치 파일** 생성 |
| 2 | 생성된 exe 파일명이 매니페스트의 `fileName`과 **완전히 동일**한지 확인 |
| 3 | 해당 파일을 서버의 **`03_license_server/downloads/`** 에 업로드 |
| 4 | 서버 `.env` 또는 `server.js`에서 **`CLIENT_APP_VERSION`**, **`CLIENT_APP_FILE`** 이 실제 파일과 일치하는지 확인 (코드 폴백만 쓸 경우에도 `UPDATE_MANIFEST`와 파일명 일치 필요) |
| 5 | `PUBLIC_BASE_URL`이 외부에서 접근 가능한 **공인 URL**인지 확인 (방화벽·4300 포트) |
| 6 | 브라우저 또는 `curl`로 `GET /api/updates/client/latest` 호출해 `installerReady: true`, `sha256` 64자 확인 |

**주의:** 다운로드 URL이 패키징된 앱에서 허용되려면 `main.js`의 **`assertHttpsDownloadUrl`** 규칙(동일 라이선스 서버 호스트의 HTTP 허용 등)을 만족해야 함.

## 7. Git / 배포 관련 주의

- **`git add .`** 시 로컬에 큰 바이너리(예: 수백 MB ZIP)가 있으면 GitHub 푸시가 거절됨. 패치 커밋은 **의도한 소스/설정 파일만** 선택적으로 `git add`하는 것이 안전.
- 민감 정보·대용량 설치 파일은 **Git LFS 또는 서버 `downloads/` 직접 배포**로 관리.
